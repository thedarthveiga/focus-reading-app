# Skill: logging

Use this skill whenever you are adding or modifying any code that handles HTTP requests, use cases, or external adapter calls. Apply the correlation ID pattern consistently across the entire request chain.

## Trigger

Invoke automatically when the user says things like:
- "adiciona log em X"
- "precisa logar Y"
- "adiciona correlationId"
- "implemente logging"
- "rastreabilidade de requests"

## Rules

### Correlation ID — origin and propagation

- The `correlationId` is **never generated by the API** — it is **always received from the client** via the `X-Correlation-Id` header on every request.
- If the header is absent, **reject with HTTP 400** immediately:
  ```json
  {
    "error": "Missing required header: X-Correlation-Id",
    "code": "MISSING_CORRELATION_ID"
  }
  ```
- The received `correlationId` is propagated through the entire chain:
  `controller → use case → ports → adapters`
- It must be included as a **required field** in all use case input DTOs and driven port contracts.
- It must be **echoed back** in the response header: `X-Correlation-Id: <same value received>`.

### Logger

- **Never use `console.log`** — always use the Pino logger already configured in the project.
- Every log message must follow this template:
  ```
  "ClassName.methodName - correlation_id: <id>"
  ```
- The `correlationId` must be present in **every log entry** within that request's scope.

### Log levels

| Level | When to use |
|---|---|
| `info` | Entry and exit of each relevant method (controller, use case, external adapters) |
| `debug` | Internal details (parameters, intermediate results) |
| `warn` | Unexpected but recoverable situations (song not found on Spotify, fallback used) |
| `error` | Caught exceptions — always with stack trace and correlationId |

### Where logging is mandatory

- **Controller:** log the incoming request (method, path, correlationId read from header).
- **Use case:** log start and end of execution with correlationId.
- **External adapters (Spotify, Claude API):** log before and after each HTTP call.
- **Every `catch` block:** log with correlationId + stack trace.

## Code patterns

### Controller (reads from header — never generates)

```typescript
const correlationId = request.headers['x-correlation-id'] as string;
if (!correlationId) {
  return reply.status(400).send({
    error: 'Missing required header: X-Correlation-Id',
    code: 'MISSING_CORRELATION_ID'
  });
}
logger.info({ correlationId }, 'ReadingSessionController.prepareSession - request received');
const result = await useCase.execute({ ...input, correlationId });
logger.info({ correlationId }, 'ReadingSessionController.prepareSession - request completed');
reply.header('X-Correlation-Id', correlationId);
return reply.status(201).send(result);
```

### Use case

```typescript
logger.info({ correlationId }, 'PrepareReadingSessionInteractor.execute - started');
// ... business logic ...
logger.info({ correlationId, estimatedMinutes }, 'PrepareReadingSessionInteractor.execute - completed');
```

### External adapter

```typescript
logger.info({ correlationId }, 'SpotifyAdapter.createPlaylist - calling Spotify API');
// ... HTTP call ...
logger.info({ correlationId, playlistId }, 'SpotifyAdapter.createPlaylist - playlist created');
```

## Constraints

- Never generate a `correlationId` inside the API layer — it must come from the client.
- Never omit `correlationId` from a log entry when it is available in scope.
- Never use `console.log`, `console.warn`, or `console.error` anywhere in `src/`.
- The `correlationId` field must be added to **all** use case input DTOs and driven port method signatures that are part of a request flow.
